Brightcove OnceUX Implementation Guide

Last Revision Date: March 2013

Overview

This guide provides the necessary information to implement the Brightcove OnceUX ad service within your workflow to obtain an ad server response in the form of a usable OnceVOD URL. OnceUX is an add-on for OnceVOD, and this document assumes that you are already familiar with the contents of Getting Started with OnceVOD.

Syntax

The OnceUX is a pre-configurable web method, based on Brightcove OnceVOD technology that allows you to make a content request and receive a single normalized response, including all ad information, in a defined XML Schema; along with your video content and in-stream video ads.

Information Gathering

Several pieces of information are needed to take advantage of the OnceUX URL construct. Resulting Once URL(s) within responses will follow OnceVOD convention for {deliveryType} and {RequestedFileType} based on the OnceUX request syntax decisions you make as listed below. (See OnceVOD Implementation Guide for more information)

  • Delivery Type: Decide if your media will be 'stitched' for progressive download, 'adaptive' for HTTP streaming or 'od' (on demand) to be used with Auto Links
  • Requested File Type: Choose from 'mp4', '3gp', 'm3u8' or 'auto' which determines delivery method according to client capabilities
  • DomainGUID, ApplicationGUID, MediaItemGUID: Each are obtained by contacting oncesupport@brightcove.com
  • Query String Parameters: Use the following pages to determine your parameters

Some of the query string parameters will be passed through and appended to the resulting
Once URL while others will be used by OnceUX when making calls to ad servers.

VAST Ad Proxy Method

This method behaves as an ad server proxy. Using the OnceUX URL construct that is pre-defined to elicit a VAST response, you would create a URL that utilizes pre-configured ad logic organized to make a request to a third party ad server. The response is normalized to a VAST 3.0 or SmartXML document. VAST XML contains a location called <mediafile> where the URL for the actual media file is held. The system parses the VAST 3.0 document for a foriegnkey or mediaitemGUID within <mediafile> to determine if it has already been ingested into the Brightcove File System. If the URL exists, it is replaced with a Brightcove Once URL that is the Ad Asset and the response is returned to the requesting client. However, if the file is not currently in the Brightcove File System, ingest and processing is initiated via Ad Cache (see the Brightcove Once Technical User Guide for more information on Ad Caching). An empty VAST 3.0 response is returned until the ad video is ready for delivery via Ad Cache.

Therefore, if the application is a flash player, iPhone app, Android app, etc., you have a searchable VAST response to utilize however you wish. In addition, you have a Brightcove Once URL that utilizes OnceVOD Automatic Device Detection technology for the ad asset.

This method also supports using HTTP cookies to handle tracking session for ad frequency capping.

VMAP Ad Aggregation Method

With this method, OnceUX retrieves a multiple ad playlist from values that are derived by summarizing or aggregating the ad viewing data based on your application (player, iPhone, etc.) from appropriate ad servers. This OnceUX URL template is pre-defined to elicit a VMAP 1.0 response providing control over ad inventory insertion where you don't control the player application.

You will create a Once URL that sends a request to Brightcove servers for content. The data returned is a list of all the ads for that content viewing session normalized into VAST 3.0 and then embedded into a VMAP 1.0 XML object. This VMAP object contains a Brightcove Once XML object, which includes Brightcove Once data and a URL. See the sample response example.

NOTE: Be aware that the VAST 3.0 object for each creative in the VMAP object will be missing a download link because the URL provided in the Once XML object will incorporate the ad resources at the correct locations causing the video ad creative content URL to be unnecessary.

The greatest difference between these two methods is that, in addition to the Once URL, the Ad Aggregation method also provides metadata and a VMAP object containing the ad load for the entire asset. This gives you greater client side flexibility with advertising while still using Brightcove Once platform. For instance, you can decide to manage tracking client side or through Brightcove Services, include banners and click-throughs not supported by Brightcove, etc.

Brightcove OnceUX utilizes a proprietary URL syntax based on OnceVOD technology. You configure it through a programmatic URL generation method customized for your specific application.

Segment variables

The Ad Proxy method sends an ad call using pre-configured ad logic, normalized and returned to the client with a Once URL for delivery.

The VMAP method implements the Ad Server Aggregator behavior, in which content is requested and a fully-formed response is given with a OnceUX URL, metadata and a VMAP object containing the ad load for the entire asset.

The URL segment variables described below are consistent between both methods with the only difference being the RequestedFileType parameter in the query string.

Optional variables within the following URL syntax examples are represented with brackets ( [ ] ).

Serving Content Variables

There are four acceptable DeliveryType/RequestedFileType combinations for the OnceUX service; od/auto, adaptive/m3u8, stitched/3gp and stitched/mp4. Brightcove Once automatically delivers one or multiple bit rate profiles based on device detection; however, you can force video to play through certain profiles by temporarily overriding the standard profile list.

NOTE: MP4 and 3GP file types will hit all impressions/tracking event URLs at once so if you are tracking hit data, it's detailed by asset rather than by ad. However, m3u8 file types have playlist associated so each URL is hit as it is viewed providing tracking granularity.

Server Side Beaconing

UMBEPARAM{key}={value}

This parameter adds beaconing-related data to the OnceUX URL so that when your content is played the appropriate third-party beaconing events are fired to your beaconing server/service. A beaconing server base URL is associated to an Application GUID within the Brightcove database and therefore not viewable. You can, however, dynamically insert values to the beaconing URLs via the query string. The beacon fires on a playback start event. Each variable in a beaconing URL can be specified according to preferences and passed dynamically to the beacon call via the query string parameter UMBEPARAM, statically mapped and inserted within the UM system, dynamically inserted via DAP (dynamic ad parameters) or any combination of these 3 options.

URL Map: /…}/{domainGUID}/[{VirtualFileName}?[UMBEPARAM{key}=[{value}]]]
Insert Values Dynamically at Request via UMBEPARAM

Through dynamically inserting values into the query string, any information you would like passed to the beaconing server that we do not already have on our system (static mapping or DAP) will be sent.

Example Key/Value pairs:
  • video=true
  • csid=sports

Beaconing URL Example

The beacons will fire at the same instance the 'Play' metric is recorded with safeguards in place to prevent multiple 'play' events skewing the analytics. Once server-side beaconing can be configured to fire upon ad start, ad complete, content start, content complete, and one of these two: content quartiles or every-N-second heartbeats. Each of those events can fire the same or different URLs, depending on what has been preconfigured. Safeguards are in place to prevent multiple beaconing events from skewing the analytics.

The static key/value pairs in the database provide values that can be extended with runtime values by utilizing UMBEPARAM and DAP if desired. If the values are not passed in via the UMBEPARAM, the values will be pulled from the mapping database. This behavior is in place for all three handlers; auto, stitched and adaptive.

Beaconing URLs do not require additional variables. However, the variables must be included if you want to use the UMBEPARAM query string parameter. You may also have multiple beaconing URLs per application.

NOTE: Each Key/Value pair specified in the UMBEPARAM must be present in the database and assigned to an application.

Disable Tracking Events

UMTP={integer}

UMTP is used to override or disable the Brightcove server side ad video impression and event tracking default behavior.

  • UMTP=0

By default, Brightcove fires linear ad video impression and tracking events to the ad server on behalf of the client/application. If the parameter is explicitly set, UM does not fire the impression or tracking event.

You are able to make requests utilizing the Foreign Key as a query string parameter rather than in the URL path. This function allows for the inclusion of characters that would otherwise be disallowed in the URL path, including unsafe characters.

Ad Frequency Capping

Umsessionkey={string}

Umsessionkey is used to store sessions for ad frequency capping for devices that don't use cookies. By using it as a query string parameter, you can control how often customers see pre-roll ads. The current default is a user will see 1 ad per X number of play starts. This parameter is not required for all Once URLs and must be configured by Client Services.

Frequency capping can be set up per application GUID and functions through the use of cookies. The function affects the behavior of server side ads, yet will only affect the pre-roll ad segment.

If your delivery and application model does not utilize cookies you may override the behavior. The value of this parameter should be a unique identifier, preferably in the form of a GUID, which your company would create and supply to UM. This unique ID is subsequently used to create a unique viewer session to enable the frequency capping behavior.

There are two value settings that can be configured. You will need to specify both:

  1. How many plays per ad
    1. Defines how many play starts a user should have per 1 ad start during a session
  2. Session time to live (TTL) in seconds
    1. The TTL associated with plays per ad defines the max duration of the viewer session before the plays per ad frequency is reset as though it were the first play

NOTE: Users playing in a frequency capped application will ALWAYS see an ad on the first play.

output={integer}

This Key-Value pair controls the object format of the response Brightcove returns.

  • XML=0
  • JSON=1
  • JSONP=2

NOTE: When using output=2, the 'callback' parameter is always required.

Type

type={integer}

This Key-Value pair controls the type of the response Brightcove returns.

  1. VAST=0
  2. SMART XML=1

NOTE: This Parameter is ONLY Applicable with the Ad Proxy Method.

Callback

callback={string}

This Key-Value pair is the callback method name to use when writing out the JSONP response.

NOTE: System default is '?output=0&type=0'. If those are the desired format and type it is not necessary to add parameters to the query string.
