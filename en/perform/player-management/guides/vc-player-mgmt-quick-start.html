Quick Start to Player Management

In this quick start you will perform the following tasks:

  • Create a Node.js application to act as your OAuth security proxy. This proxy will use your access token to make calls to the needed APIs.
  • Obtain credentials to use the Content Management System (CMS) API.

For details on the API used in this document you can view the Content Management System API Overview document and the Content Management System API reference.

Note: The steps in the document involve curl and other command-line tools that are native to Mac and Linux systems, but not part of the Windows operating system. If you are on Windows, you can get these tools by installing cygwin. During the install you will be prompted for packages to install. Open the Net package, then check the curl option and then finish the installation. Once you have done this you will be able to follow the Quick Start to the Content Management System successfully as written using the cygwin command line. Tools are available to run curl at the normal Windows command line, but not detailed in this document.

Create proxy

Security on the Player Management API uses OAuth2. This requires some work to access the API.

  1. In this quick start it is recommended you set up a Node.js application to act as a proxy. The OAuth Quick Start guides you through the steps to create a basic proxy in Node.js that gets an access token for a Brightcove API, makes the API call, and then returns the response data.
  2. You will also need a local web server running to serve up a testing page, tester.html, which works with the proxy. Note that simply browsing the page as a file will NOT work as the page utilizes an iframe, which requires an HTTP server.

Obtain credentials

To use the CMS API you will need proper credentials. Later in this quick start, you will use the credentials in a form when submitting requests to APIs.

  1. To obtain proper credentials use your choice of these two options (Be sure to read the paragraph following the bullets to ensure you ask for the proper operations):
  • OAuth: Get Client Credentials Using CURL
  • OAuth: Get Client Credentials Using Postman

Whichever option you choose, you will need to ask for the correct operation permissions. In this quick start you will need both video and ingest-profile operations. The following can be used with curl or Postman to get the proper permissions:

Create player

Of course part of the Player Managment API is to create a Brightcove player. This player will play a Video Cloud video. You will use an HTTP POST method and pass in the player name and Video Cloud video that will play.

  1. You will now use the proxy in conjunction with the tester.html page to create a player and configure the Video Cloud video to be played. This configuration information will be passed to the API as a JSON object. Browse to tester.html and complete the form as follows:
Form Field Value
Client id Returned in your client credentials
Client secret Returned in your client credentials
Call type POST
Call body {"name":"**YOUR_VIDEO_NAME","configuration":{"video_cloud":{"video":"**VIDEO_ID**"}}}
Full request URL for the API call https://players.api.brightcove.com/v1/accounts/**YOUR_ACCOUNT_ID**/players

  1. View your video by using one (or all) of the following values:

Update player

You can change a player you have created. In this section you will implement a very simple update, replacing in the player one Video Cloud video with another. You will use an HTTP PATCH method and pass in a new Video Cloud video. You will see more complex update update use cases later in this document.

Note: In this case you are using an API call to update the video. In the Dynamically Changing the Video document, you see an example of changing the video on a user event.

  1. You will now use the proxy in conjunction with the tester.html page to update the player to play a new Video Cloud video. This new video information will be passed to the API as a JSON object. Browse to tester.html and complete the form as follows:
Form Field Value
Client id Returned in your client credentials
Client secret Returned in your client credentials
Call type PATCH
Call body {"video_cloud":{"video":"**VIDEO_ID**"}}
Full request URL for the API call https://players.api.brightcove.com/v1/accounts/**YOUR_ACCOUNT_ID**/players/**YOUR_VIDEO_ID**/configuration

After you click the Submit button, the text area following the button will be populated with JSON object containing information on the newly updated player, as shown in the following screenshot.

  1. Use either the preview_url or preview_embed_code values to see the changes in your player's configuration. You will see the image (poster) displayed before you start the video playing.
  2. Now you'll publish the player to push it to the correct location for users to view. Submit the form with the following:
Form Field Value
Client id Returned in your client credentials
Client secret Returned in your client credentials
Call type POST
Call body
Full request URL for the API call https://players.api.brightcove.com/v1/accounts/**YOUR_ACCOUNT_ID**/players/**YOUR_VIDEO_ID**/publish

After you click the Submit button, you will have players that differ from the preview versions. The published versions differ from the preview versions in these ways:

  • The JavaScript and CSS are minified, concatenated and inlined into the player directly.
  • A low resolution version of the poster image is generated and inlined into the page to improve perceived load times on networks with a high request-setup time (i.e. cellular data networks).
  • The previous version of the player is saved so it can be recovered if problems are discovered after the update goes live.

Note: On player creation (step 4 above) the player is also automatically published. That is why on player creation the JSON response included both published and preview information.
