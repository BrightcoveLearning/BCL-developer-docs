Stream Concurrency

This is a temporary STAGING document for review only. For the official Brightcove documentation site, go to support.brightcove.com

In this topic, you will learn how to implement stream concurrency when using Brightcove's Playback Authorization Service.

Introduction

Stream concurrency allows a customer to limit the number of videos a specific user can watch at any given time. This feature hooks into the DRM workflow so it is secure and can be used with minimal integration.

If you are new to this feature, see the Overview: DRM with Playback Authorization Service document.

Implementation

To use stream concurrency with Brightcove's Playback Authorization Service, follow the steps in the Implementing DRM with Playback Authorization Service document.

Be sure to include the JWT token claims required for concurrency.

Behavior examples

The following examples demonstrate different strategies for limiting playback streams.

General Behavior

For DRM systems supporting renewal requests, DRMtoday uses a defensive limiting strategy, where the oldest session is stopped first, once its upcoming renewal request is processed. This has the advantage of always granting the most recent user request vs. denying new requests because other playbacks are believed to be in progress, but not known with certainty. (e.g. a device or network failure).

User activity:

  1. Amy, Ben and Claire enjoy watching a movie.
  2. Dave decides to also start watching, causing Amy's upcoming renewal request to be denied.
  3. Amy restarts the movie, now causing Ben's renewal request to fail.
  4. Ben starts watching another movie some time later, causing Claire's movie to stop.

PlayReady Only

Microsoft PlayReady does not support renewal requests, but uses secure stop instead. The only way to limit the total number of streams is to deny any new sessions.

User activity:

  1. Amy, Ben and Claire enjoy watching another movie.
  2. Dave decides to also start watching, but his request is denied due to the limit.
  3. Amy stops watching and her player sends the secure stop request. This opens a new slot.
  4. Dave can now use the new slot.

PlayReady: Secure Stop

In case the secure stop signal is not received, the session duration limit applies.

User activity:

  1. Amy, Ben and Claire enjoy watching a movie.
  2. Amy's device is crashing and not sending the secure stop request.
  3. Dave also wants to watch, but his request is still denied.
  4. The session duration is reached and the slot is available again.
  5. Dave can now watch his movie.

PlayReady interaction with other DRM types

When the maximum number of sessions has been reached by PlayReady requests, even Google Widevine and Apple FairPlay license requests are denied and don't lead to new sessions.

User activity:

  1. Amy, Ben and Claire enjoy watching a movie.
  2. Dave decides to also start watching, causing Ben's movie to stop. He's using a device supporting renewal requests and is upcoming request gets denied.
  3. Ben decides to switch to another device supporting secure stop, causing Dave's next renewal request to fail. Now, all slots are occupied by devices with secure stop.
  4. Dave has no free slot to watch.
  5. Amy stops watching, which frees a slot for Dave to use.

Limitations

The following limitations apply with the initial release of this feature:

  • For HLSe (AES-128), the token will be appended to the master manifest request, and will propagate to the encryption key URL. Because these URLs are appended to the master manifest, customers should consider restricting the number of uses to minimize exposure of their content.
  • For HLSe (AES-128), players will make mutlitple requests when playing a video, typically one per rendition. The maxu claim needs to be set high enough to account for these requests. You should consider additional claims to minimize the exposure of your content. Switching renditions will request a new license.
  • Tracking of the maxu and maxip claims is eventually consistent. This means that if a token using these claims is reused many times in quick succession, it is possible to see that token accepted more times than those claims declare. The tracking becomes consistent within a few seconds and after that window the token will be blocked as expected.
