Implementing Concurrent Stream Limits

This feature is available for a specific set of customers with access to the Limited Availability phase of the Playback Rights Management Service.

In this topic, you will learn how to implement concurrent stream limits when using Brightcove's Playback Authorization Service.

Introduction

With concurrent stream limits, you define the number of videos a specific user can watch at any given time. Limiting concurrent streams per user discourages viewers from sharing their credentials with friends who donâ€™t have accounts.

If the viewer, or someone with their account credentials, tries to open an additional stream while the maximum number of streams is in use, the player stops the oldest stream when a renewal request occurs.

This feature hooks into the DRM workflow so it is secure and can be used with minimal integration.

Account setup

Contact your account manager to have your account enabled for stream concurrency.

Implementation

Using stream concurrency follows the same steps as in the Implementing DRM with Playback Authorization Service document.

To enable stream concurrency, you must include the JWT token claims required for concurrency.

Renewal Interval

Mid-stream playback authorization requests need to specify a concurrency limit in the JWT token along with a public key id.

Today each renewal request counts as an additional license that would go towards the number of your purchased licenses. This should be taken into account when you are deciding on an interval.

General Behavior

Brightcove supports limiting playback streams for DRM-protected content. The default behavior is to stop the oldest session when a new renewal request is processed. This allows new user requests to be given access, while blocking the oldest requests which may no longer be valid due to a network or device failure.

User activity:

  1. Ben, Jason and Sarah begin watching a movie.
  2. Jamie decides to also start watching, causing Ben's upcoming renewal request to be denied.
  3. Ben restarts the movie, now causing Jason's renewal request to fail.
  4. Jason starts watching another movie some time later, causing Sarah's movie to stop.

Using PlayReady

When using PlayReady, you should expect the same general behavior as defined in the previous section. However, you need to implement a manual renewal in your players as described in Microsoft's Concurrency Limiting document.

Notes

The following notes apply to this feature:

  • Manual renewal is required for any DRM other than Widevine. (PlayReady & FairPlay).
  • The Brightcove Native SDK for iOS/tvOS handles FairPlay renewals automatically.
  • Non-Brightcove players need to request a new license at an interval less than the renewal period.

Limitations

The following limitations apply to this feature:

  • Concurrency limit support is not yet supported in Safari due to the browser's limitations.
