CMS API Overview

The CMS API provides uncached read access to the data. This is important for time-sensitive publishing workflows because you make a changes to videos and playlists using the CMS API and quickly read the data to verify that is correct.

Note: The CMS API is intended for integrations and publishing workflows. You should never make client-side calls to this API from a public web page, as doing so would involve exposing your client credentials for the API.

Note: The CMS API methods never return deleted videos. The 'state:DELETED' clause will be ignored by the CMS API whether you're asking for a count or for the videos themselves. CMS API Notifications will inform you when videos are deleted (or other changes to videos are made).

If you are going to be retrieving all your videos or large sets of videos, be sure to see the note on large data sets.

General Information

Base URL

The base URL for the CMS API is:

https://cms.api.brightcove.com/v1

Account path

In all cases, requests will be made for a specific Video Cloud Account. So, you will always add the term accounts followed by your account id to the base URL:

https://cms.api.brightcove.com/v1/accounts/{account_id}

Authentication

Authentication for requests requires an Authorization header:

Authorization: Bearer {access_token}

The access_token is a temporary OAuth2 access token that must be obtained from the Brightcove OAuth service. For details on how to obtain client credentials and use them to retrieve access tokens, see the Brightcove OAuth Overview.

Operations

When you request client credentials, you will need to specify the type of account access or operations that you want. The following is a list of the currently supported operations for the CMS API:

  • Video data:
    • video-cloud/video/read
    • video-cloud/video/create
    • video-cloud/video/delete
    • video-cloud/video/update
    • video-cloud/video/all
  • Playlist data:
    • video-cloud/playlist/read
    • video-cloud/playlist/create
    • video-cloud/playlist/delete
    • video-cloud/playlist/update
    • video-cloud/playlist/all
  • Notifications:
    • video-cloud/notifications/all (for Notifications)

Rate limiting

This CMS API provides uncached read access to the data. This is important for time-sensitive publishing workflows because you make a changes to videos and playlists using the CMS API and quickly read the data to verify that is correct.

The CMS API is not appropriate for high scale runtime usage (such as accessing videos on a high traffic public web page). For high traffic applications, you must use a cached interface such as : the Media API, Gallery, Players, or the Native SDK's.

To ensure the performance of the Video Cloud system, no more than 5 concurrent calls to the CMS API are allowed per account. Access frequency should be less than 10 queries per second.

If multiple applications will be integrating to the CMS API for an account, then these applications should have back-off and retry logic to handle instances where concurrency limits or rate limits are reached.

Currently the API supports the following request types:

  • GET
  • POST
  • PATCH
  • PUT
  • DELETE

Parameters

Note that all parameters are optional. Except where noted, they apply to GET calls for videos and playlists.

Parameter Description
q Query string for searches - see Search syntax for details
limit Number of videos to return - must be an integer between 1 and 100. Default: 20
offset Number of videos to skip (for paging results). Must be a positive integer. Default: 0
sort A string that specifies the field to sort by. Start with - to sort descending. If a value for q is provided, then the default sort is by "score" (relevance of the search results to the original query). If no value for q is provided, then the default sort is by updated_at descending.
playable Available at the /videos endpoint. Set q=playable:true to return playable videos and set q=playable:false to return unplayable videos. Default: true

Search videos

Brightcove's CMS API provides a programmatic way to search for videos in your Video Cloud library.

To perform basic and complex searches on your video data, you will use the q parameter:

https://cms.api.brightcove.com/v1/accounts/921483702001/videos?q={search terms}

For details on how to search for videos, see the Using the CMS API: Search Videos document.

For playlists, the supported values for the search string are limited. You can currently only search by tags. Here are some example valid searches:

  • tags:foo
  • +tags:bar
  • tags:foo,bar
  • +tags:"foo bar","boo yah"

For specific scenarios:

  • contains all: "+tags:foo,bar"
  • contains any: "tags:foo,bar"

Paging results

Use the limit parameter to specify how many items you want to return on a request - up to 100. You can then use the offset parameter to page through result sets that are larger than the limit. The offsetis the number of items to skip. For example, for a search like this:

/videos?q=updated_at:2014-01-01..2014-06-30&limit=25&offset=50

would return videos 51-75 of the total result set, assuming the total result set has at least 75 videos.

limit and offset can be used for both videos and playlists.

Sorting results

Use the parameter sort=field_name (or sort=-field_name for descending order)to specify how the results should be sorted - you can sort both video and playlists. You can sort on the following video fields:

  • name
  • reference_id
  • created_at
  • published_at
  • updated_at
  • schedule_starts_at
  • schedule_ends_at
  • state
  • plays_total[1]
  • plays_trailing_week[1]

Notes

  • [1] You can sort on plays_total or plays_trailing_week, but these fields are not included in the results

Sorting Results

You can sort playlists on the following fields:

  • name
  • created_at (default)
  • updated_at
  • reference_id

If you do not provide a sort value for a video search call, the results will be sorted by relevance. If you do not provide a sort value for a list call, the results will be sorted by updated_at descending.

All videos and large data sets

If you are retrieving all the videos in your account, or a large number of videos, there are some things you must be aware of:

  1. You may be tempted to use the largest allowed page_size (100), but it's better to retrieve videos in batches of 25 or less to minimize the possibility of API requests timing out
  2. As you are paging through large data sets, it is possible that the video data may be updated during the operation, which could cause items to shift in responses:
    • You might see an item repeated on successive pages
    • An item might be missed, as it has shifted to a previous response set

    To account for the first possibility, your app should de-dupe the complete item list after you have finished retrieving videos. To handle the second possibility, you need to compare the total number of items retrieved (after de-duping) to the number you were expecting, and then rerun the requests, sorting the results by last_modified_date (descending) - you shouldn't need to retrieve more than one batch to pick up missed items.

  3. You can decrease the likelihood of the scenarios in the previous item by sorting the returned results appropriately. The default sorting by relevance for searches is based on a complex algorithm that looks combinations of keywords, tags, and custom field values. If you are searching for videos based on multiple keywords, tags, and/or custom fields, sort by relevance is exactly what you want. However, if you are just trying to retrieve all or a large set of your videos, setting the sort_by parameter explicitly will give you more control over the order of the returned items.

Update video

Use the HTTP PATCH request to update properties for a video object in your account.

For an example of updating a video, see the Quick Start To CMS document.

Check the data type

When updating properties, keep in mind that with arrays you need to include ALL the array values. This is because arrays are all-or-none. If you had 3 array values and try to update just 1 value, the resulting array will only contain 1 value. If you had 3 array values, you must include all 3 entries, including the updated value.

Updating properties based on data type:

  • Non-array types: With properties like description, name and reference_id, you can update each field without affecting the others.

    Example: If you have a description and long_description defined, you can update one without affecting the other.

  • Map types: With properties like geo, schedule, link, you can update each sub-field without affecting the other.

    Example: If you have a link.url and link.text defined, you can update one without affecting the other.

  • Array types: With properties like cue_points and tags, you have to include all the items in the array, and not just the one you are updating.

    • Example: With cue_points, you have to include all cue points and not just the one you are updating. If you include just the cue point you are updating, you will end up with just that one cue point.

Delete video

You can delete a video object in your account.

Note: You will not be able to delete the sharer copy (orignal) of a shared video. You can delete shared copies.

The fundamental call information for deleting a video object is:

  • HTTP method: DELETE
  • Endpoint: https://cms.api.brightcove.com/v1/accounts/{account_id}/videos/{video_id}

Video Playlist References

The videos references endpoint allows you to get a list of ids for the manual playlists that a video belongs to (using a GET request) or remove the video from all manual playlists (using a DELETE request). The latter operation must be performed before the video itself can be deleted (if it is included in any manual playlists).

Get the manual playlists that a video belongs to:

  • HTTP method: GET
  • Endpoint: https://cms.api.brightcove.com/v1/accounts/{account_id}/videos/{video_id}/references

Remove the video from any manual playlists that it belongs to:

  • HTTP method: DELETE
  • Endpoint: https://cms.api.brightcove.com/v1/accounts/{account_id}/videos/{video_id}/references

Notifications

You can receive notifications when video_change events occur in your video library. Notifications will be sent to the URL you specify, which should point to an application capable of handling HTTP POST requests. You can specify up to 10 endpoints to receive notifications by making a POST request to https://cms.api.brightcove.com/v1/accounts/{account_id}/subscriptions and including JSON in the request body like this:

{
    "endpoint":"http://solutions.brightcove.com/bcls/di-api/di-callbacks.php",
    "events":["video-change"]
}

Notifications are sent in JSON format. Here's an example:

{"timestamp":1423840514446,
    "account_id":"775205503001",
    "event":"video-change",
    "video":"4020894387001",
    "version":26}

Requests to create a subscription will receive an HTTP 422 error response for the following conditions:

  • The endpoint or events field is missing from the request body
  • The events field value is not a list (array)
  • the subscription defined already exists
  • you already have 10 subscriptions to this event

Notification failures

The notification system treats any 4xx or 5xx return from the customer server as a retriable failure. Failing callbacks will be retried up to 20 times, with an exponentially increasing delay between subsequent callbacks. The first few retries will happen within minutes of the initial callback attempt. If the callback continues failing, and we get all the way out to the 20th retry, the retry delay will be a few days.

Note: video_change events are triggered by automated system processes as well as user actions, so these events will not always corresponds to changes you made to video properties in Studio or via the APIs.

Firewalls

In case your organization has a strict policy regarding sources of incoming traffic through your firewall, the following fixed IP addresses are used to send notifications:

  • 54.84.92.184
  • 54.175.126.166
  • 54.164.120.55

These values are subject to change in the future. The IPs are within the AWS ec2 us-east-1 address space and we anticipate that future IP addresses will also be, if you don't want to deal with specific IP address changes, you can configure your firewalls and routing rules to allow traffic from any of us-east-1.

Endpoint for notification subscriptions

/accounts/{account_id}/subscriptions

Create a new subscription

To create a new subscription, submit a POST request with a request body including the endpoint that you want notifications delivered to and video-change as the single item in an events array:

{ "endpoint": "http://solutions.brightcove.com:9002", "events": [ "video-change" ] }

Get a list of your subscriptions

To get a list of all your subscriptions, send a GET request to the subscriptions endpoint.

Get or Delete a single subscription

Endpoint

/accounts/{account_id}/subscriptions/{subscription_id}

A GET request will retrieve the subscription. A DELETE request will delete the subscription. You cannot update a subscription at this time. If you need to modify a subscription, you will need to delete it and create a new one.
