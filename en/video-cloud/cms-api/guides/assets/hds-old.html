This topic explains Video Cloud support for Akamai HDS (also known as HD2) remote assets. Video Cloud also supports HDS for ingested assets.

Support for HDS streaming

  • Addition of HDS remote assets is supported via Video Cloud Studio and the Media API
  • The following asset types are currently supported:
    • VOD (unencrypted) - F4V, MP4, SMIL, CSMIL formats
    • VOD (encrypted) - MP4, SMIL, CSMIL formats
    • Live (unencrypted)
    • Secure Live streams

Limitations

Brightcove plans to support the following features in future releases:

  • Preview from the Media Module Details Pane, or the Video Files editor
  • Image Capture

There are no plans to support the following:

  • Player verification

Workflow for creating HDS remote video assets

Below are some special considerations for creating HDS VOD and live streaming assets.

NOTE: if you plan to use the Brightcove native player SDKs to create video-enabled apps for iOS and Android, you MUST use the method explained in https://support.brightcove.com/en/video-cloud/docs/creating-videos-remote-assets-using-media-api

VOD assets

For URLs to SMIL files, enter the URL including the SMIL extension, do not use the any path information past the SMIL extension, if it exists. For example, in the following URL:

http://zerihdndemo-f.akamaihd.net/z/h264/robinhood/robinhood.smil/manifest.f4m

use the following URL in creating your remote asset in Video Cloud via Studio or the Media API:

http://zerihdndemo-f.akamaihd.net/z/h264/robinhood/robinhood.smil

For URLs containing MP4 or F4V extensions, if the URL contains all of the rendition information in a single URL, the entire URL including the SMIL (or CSMIL extension) can be used to create a remote video through the Media Module or Media API. All path information past the SMIL/CSMIL extension can be discarded. For example, a URL like this:

http://multiplatform-f.akamaihd.net/z/multi/april11/hdworld/
  hdworld_,512x288_450_b,640x360_700_b,768x432_1000_b,1024x576_1400_m,
  1280x720_1900_m,1280x720_2500_m,1280x720_3500_m,.mp4.csmil/manifest.f4m
the URL should be trimmed to exclude everything past the .csmil extension.

However, this is not the optimal use case, since asset URLs support a maximum of 250 characters. Using a single URL with information for several renditions can exceed this limit, and will throw an error. Using a single URL also creates a single asset record, and may affect rendition selection.

The optimum way to create a video with multiple renditions is to use individual asset URLs to create remote renditions. If the only URL available contains all rendition information (as in the example above), the URL can be edited into separate rendition URLs.

Here is the typical procedure. Assume a video URL like this:

http://multiplatform-f.akamaihd.net/z/multi/april11/hdworld/hdworld_,
512x288_450_b,640x360_700_b,768x432_1000_b,1024x576_1400_m,
1280x720_1900_m,1280x720_2500_m,1280x720_3500_m,.mp4.csmil/manifest.f4m

Do the following to create the individual rendition URLs:

  1. Use the URL up to the first comma as the URL base
  2. Append the rendition information (for example, the “512x288_450_b” in the example above) and the video file type (in the example above, “.mp4”) to the URL base for the first rendition. The first rendition, from the example above, would be:
    http://multiplatform-f.akamaihd.net/z/multi/april11/hdworld/hdworld_,512x288_450_b.mp4
  3. Repeat the step 2 for each rendition described in the URL. In the example above, Seven renditions would be created for the remote video

Live Video

For a URL formatted like this: http://multiformatlive-f.akamaihd.net/z/demostream_1@2131/manifest.f4m, use the URL up to the “manifest.f4m” file path to configure the live stream. In the example above, the live stream URL to use in creating the remote video would be:

http://multiformatlive-f.akamaihd.net/z/demostream_1@2131

The (live feed) stream name for encoders should follow the format of [EVENT_ANGLE_BITRATE]@StreamID while [EVENT_ANGLE]@StreamID should be used as the playback URL in remote video assets.

Note that the frame size and bitrates are not required, as those are detected by Akamai and are put in the generated manifest file automatically. Be sure your final stream includes the /z/ directory and not the /i/ directory so our player can select the appropriate streams.
