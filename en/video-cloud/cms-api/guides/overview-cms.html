CMS API Overview

The CMS API is currently in beta and is not supported by Brightcove Support. If you would like access to the beta, please contact your Brightcove Account Manager.

Brightcove's CMS API provides a programmatic way to create, update and access data for your Video Cloud videos.

Base URL

The base URL for the CMS API is:

https://cms.api.brightcove.com/v1beta1

Account path

In all cases, requests will be made for a specific Video Cloud Account, so you will always add to the base URL the account path:

/accounts/{account_id}

Authentication

Authentication for requests requires an Authorization header:

Authorization: Bearer {access_token}

The access_token is a temporary OAuth2 access token that must be obtained from the Brightcove OAuth service. For details on how to obtain client credentials and use them to retrieve access tokens, see the Brightcove OAuth Overview.

Operations

When you use the PBI API you will perform different operations, such as reading an ingest profile and writing ingest information to your account. The following is a complete list of ingest profile operations:

  • video-cloud/video/profile/read
  • video-cloud/video/account/create
  • video-cloud/video/account/update

Methods

Currently the API supports the following request types:

  • GET
  • POST

Resources

The following resources are currently accessible via the CMS API:

Resource Endpoint Description
/videos Metadata and asset information about videos. Will list a page of videos if no video ID is given. Supports search (q param), sort, and paging (limit and offset)
/videos/{video_id} Metadata and asset information about the video with the given ID
/videos/{video_id}/sources Metadata for this video's sources (renditions)
/videos/{video_id}/images Metadata for all of this video's images: poster, thumbnail, logo overlay
/videos/ref:{video_reference_id} Metadata and asset information about the video with the given reference ID
/videos/ref:{video_reference_id}/sources Metadata for this video's sources (renditions)
/videos/ref:{video_reference_id}/images Metadata for all of this this video's images: poster, thumbnail, logo overlay

Note that all parameters are optional.

GET parameters

Parameter Description
q Query string for searches - see Search syntax for details
limit Number of videos to return - must be an integer between 1 and 100. Default: 20
offset Number of videos to skip (for paging results). Must be a positive integer. Default: 0
sort A string that specifies the field to sort by. Start with - to sort descending. If a value for q is provided, then the default sort is by "score" (relevance of the search results to the original query). If no value for q is provided, then the default sort is by updated_at descending.

Search Syntax

Search terms are specified using the q parameter. 

q={search terms}

The query string you provide should be a url encoded list of terms separated by a space (+ after encoding).

Basic search

If provide no qualifiers on the search term:

q=foo

This will execute a text search against the following fields: name, description, long_description, tags, reference_id, and all custom fields.

Limit search to a specific property

You can search against values for a specific field:

q=name:foo

This will search for videos with the term "foo" in the name field of the video. The supported fields are:

Field Legal values
name strings or quoted strings
text strings or quoted strings (name, description, long_description)
tags strings or quoted strings
reference_id string or quoted string
state ACTIVE, INACTIVE, DELETED, PENDING
updated_at date range
created_at date range
schedule.starts_at date range
schedule.ends_at date range
published_at date range
complete true or false

The fields that take a string result in a text search against that field.

Required and excluded terms

You can mark a term as required (returned videos MUST match) or prohibited (returned videos must NOT match). This is controlled with a + or - sign immediately preceding the term.

example urlencoded meaning
+foo %2Bfoo videos must include the term 'foo' in the name, description, or long_description
-foo %2D videos must NOT include the term 'foo' in the name, description, or long_description
+name:foo %2Bfoo videos must include the term 'foo' in the name
-name:foo %2D videos must NOT include the term 'foo' in the name

Quoted search terms

By default, a search term could match similar words. If you want to do an exact match, or match multiple words, just wrap the term in quotes:

"foo"
"foo bar"

This also works when searching against a specific field:

name:"foo"
name:"foo bar"

Custom fields

You may search on your custom fields similarly. Custom fields are treated as strings:

my_field:foo
my_field:"foo"

Date ranges

If the field you are searching on is a date, then you can specify a range of dates to search for:

updated_at:2012-08-01T00:00:00Z..2012-10-08T00:00:00Z

This will search for all videos with an updated_at value between Aug 1, 2012 and October 8, 2012.

Supported date formats

format meaning
2012-08-01T00:00:00Z This represents a time in UTC.
2012-08-01 This represents a midnight on a day in UTC. The example is equivalent to 2012-08-01T00:00:00Z
-1d The current time minus 1 day. (see below)
+1d The current time plus 1 day. (see below)
Relative dates

For relative dates we support a direction (+ or -) followed by a number, followed by a duration. Legal durations are: minutes, hours, days

Examples
  • -1day
  • -2days
  • +4hours
  • -60minutes
  • -1d
  • -2d
  • +4h
  • -60m
Open ended ranges

If you want to match all dates until a given time, or match all dates since a given time, leave off one end of the range. For example:

Search for all videos modified in the last 2 days
q=updated_at:-2days..
Search for all videos modified before August 11, 2013:
q=updated_at:..2013-08-11T00:00:00Z

Combining search criteria

You can combine criteria for complex searches:

q=%2Bname:gossip+%2Bupdated_at:2010-08-01T00:00:00Z..2010-10-08T00:00:00Z&sort=updated_at

Response

Create video

POST requests will create a new video object in the account. Only the object is created - this is not a method that allows you to upload a video. See the Dynamic Ingest API for information on how to send the video source file to Brightcove.

When you use POST to create a video you can set some property fields during the POST operation. Not all fields can be set during the POST though, for example cue_points and sharing. The following table lists those fields for which you can provide values when the video is created.

Field Type Required Comments
account_id String no Must match account ID in the URL
name String yes
reference_id String no
description String no
long_description String no
state String no Defaults to "ACTIVE" if not set explicitly
tags Array of Strings no
custom_fields Map of field-value pairs no
geo Map of geo properties no
geo.countries Array of country code Strings no
geo.exclude_countries Boolean no
geo.restricted Boolean no
schedule Map of scheduling properties no
schedule.ends_at String in ISO-8601 date format
schedule.starts_at String in ISO-8601 date format

Paging

With the CMS API, you can implement paging. This allows you to handle large result sets in smaller groups, which improves performance.

A maximum of 100 records will be returned from a single request to the CMS API.

For page handling, follow these steps:

  • Get a count of the total number of records in the result set.
  • Calculate the number of pages for your result set.
  • Loop through and call the CMS API for each page of data.

Getting the record count

Use the counts resource to return a total count of records in your result set.

You can get a total count of videos in your account:

https://cms.api.brightcove.com/v1beta1/accounts/921483702001/counts/videos

Results:

{
  "count" : 74
}

You can limit the count by using the optional q= parameter. Here we get the number of videos which have a tag value of wildlife.

https://cms.api.brightcove.com/v1beta1/accounts/921483702001/counts/videos?q=tags:wildlife

Results:

{
  "count" : 6
}

Calculating number of pages

To calculate the number of pages, take the total number of records in your result set and divide it by the number of records per page (page size).

JavaScript example:

var numberOfPages = Math.ceil(jsonData.count/pageSize);

Retrieving pages

Loop through the number of pages that you calculated in the previous step and call the CMS API with the current limit set.

https://cms.api.brightcove.com/v1beta1/accounts/921483702001/videos
