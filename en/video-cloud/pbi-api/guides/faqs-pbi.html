Pull-Based Ingest API FAQs

FAQs

Are API requests limited?

No

How many jobs can be processed simultaneously?

Unlimited

What are the supported input formats?

The PBI API supports 99.9% of all input formats. Virtually every significant format over the last 30 years is supported by our platform.

How fast is Pull Ingest?

Pull Ingest is a horizontally scalable, cloud transcoding solution that processes all of your content in parallel. Becauase of this, we generally see transcoding times at faster than real time. Depending on how quickly Pull Ingest can download the file from your storage location, an unlimited number of jobs can process at a highly consistent rate.

What region is Pull Ingest located in?

Currently, Pull Ingest workers are running in the AWS US East region. Any content hosted in S3 buckets in that region can be downloaded with no egress charge. Content hosted on other cloud providers or in other regions may incur an egress cost with your cloud provider. In the future, we plan to enable this technology to run in any of the current AWS regions.

Is DRM supported?

At launch, Pull Ingest will not support DRM. We plan to add DRM support by January 2015.

Why pull instead of push?

Legacy Video Cloud Ingest methods all supported some form of push - Media API, Batch FTP, and Aspera. Push works well for certain use cases, but Pull has significant advantages. Since a job request consists only of a URL to the file, any file corruption or transfer issues can be solved by automatically retrying the job. Downloading files allows us to horizontally scale the platform - instead of funneling all traffic through an FTP or HTTP post server, downloads can occur on the same worker that performs the transcoding. When files are hosted on cloud storage, the scale becomes truly astounding, a virtually limitless number of jobs can be run in parallel.

Is pull ingest secure?

Security is a paramount concern for Brightcove. Pull Ingest supports secure transport methods - sftp, https, or bucket policies with S3. By using these methods, the transfer to retrieve a source file is secured. Once in the cloud, if the decision is made not to store the master, the master file is hosted on transcoding workers and deleted immediately after the transcode is completed. File names are hashed and never revealed. The same security policies maintained by Zencoder are used for the transcode process, and in over 4 years of operations, no file has ever been compromised on the Zencoder platform.
